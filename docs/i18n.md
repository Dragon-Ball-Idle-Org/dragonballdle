# 🌍 Internationalization (i18n)

DragonBallDle supports **20 languages** using [next-intl](https://next-intl.dev/). This guide covers how translations work and how to add new languages.

## Supported Languages

| Locale | Language | Locale | Language |
|---|---|---|---|
| `ar-SA` | Arabic | `ms-MY` | Malay |
| `de-DE` | German | `pt-BR` | Portuguese (Brazil) |
| `en-US` | English (default) | `ru-RU` | Russian |
| `es-ES` | Spanish | `th-TH` | Thai |
| `fil-PH` | Filipino | `tr-TR` | Turkish |
| `fr-FR` | French | `uk-UA` | Ukrainian |
| `hi-IN` | Hindi | `vi-VN` | Vietnamese |
| `id-ID` | Indonesian | `zh-CN` | Chinese (Simplified) |
| `it-IT` | Italian | `zh-TW` | Chinese (Traditional) |
| `ja-JP` | Japanese | `ko-KR` | Korean |

The default locale is `en-US`. All routes are prefixed with the locale: `/pt-BR/classic/`, `/ja-JP/silhouette/`, etc.

## File Structure

```text
messages/
├── en-US.json    # Default / reference file
├── pt-BR.json
├── ja-JP.json
└── ...           # One file per locale
```

```text
src/i18n/
├── navigation.ts   # Wrapped Link and redirect (MUST use these)
├── routing.ts      # Locale list and default locale
└── request.ts      # Server-side locale resolution
```

## Critical Rules

> [!CAUTION]
> **Never** use `next/link` or `next/navigation` directly. Always import from `@/i18n/navigation`. This ensures links include the locale prefix.

> [!CAUTION]
> **Never** hardcode text in components. Always use translation functions.

```typescript
// ✅ Correct
import { Link } from "@/i18n/navigation";
<Link href="/classic">...</Link>

// ❌ Wrong — will break locale routing
import Link from "next/link";
<Link href="/classic">...</Link>
```

## Translation Patterns

### Server Components (default)

```typescript
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("home");
  return <h2>{t("title")}</h2>;
}
```

### Client Components

Client components cannot call `getTranslations` directly. Use one of these approaches:

**Option A: `useTranslations` (simple cases)**

```typescript
"use client";
import { useTranslations } from "next-intl";

export function GameButton() {
  const t = useTranslations("game");
  return <button>{t("start")}</button>;
}
```

**Option B: `TranslationProvider` (complex bundles)**

When a client subtree needs many namespaces, load them server-side and pass via provider:

```typescript
// Server layout
import { getTranslationsBundle } from "@/lib/client-translations";

export default async function Layout({ children }) {
  const bundle = await getTranslationsBundle();
  return (
    <TranslationProvider messages={bundle}>
      {children}
    </TranslationProvider>
  );
}
```

### Nested Keys

Message files use nested JSON objects:

```json
{
  "home": {
    "metadata": {
      "title": "DragonBallDle — Daily Dragon Ball Guessing Game",
      "description": "Can you guess today's Dragon Ball character?"
    },
    "classic": {
      "title": "Classic"
    },
    "silhouette": {
      "title": "Silhouette"
    }
  }
}
```

Access nested keys with dot notation: `t("home.classic.title")` or namespace: `useTranslations("home.classic")`.

## Adding a New Language

### Checklist

1. **Create the message file**
   ```bash
   cp messages/en-US.json messages/<locale>.json
   ```
   Translate all values (keep keys unchanged).

2. **Register the locale** in `src/i18n/routing.ts`:
   ```typescript
   export const routing = defineRouting({
     defaultLocale: "en-US",
     locales: [
       // ... existing locales
       "<locale>",  // Add here alphabetically
     ],
   });
   ```

3. **Verify** — start the dev server and navigate to `/<locale>/`.

4. **Test** — ensure all pages render without missing translation warnings.

> [!TIP]
> Use `en-US.json` as the reference file. Every key present in `en-US.json` must exist in the new locale file.
